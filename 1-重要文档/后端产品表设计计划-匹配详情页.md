# 后端产品表设计计划 — 匹配产品详情页

> 目标：让 Strapi 产品内容类型与前端「产品详情页」展示需求一一对应，支持在后台维护详情区全部可配置内容。  
> 可在现有架构上**增量优化**，也可**重新设计一张表**（旧数据需迁移时再考虑）。

---

## 一、产品详情页需要的数据清单

从前端 `ProductDetailClient` 与路由约定，详情页用到的产品相关数据如下。

| 区块 | 字段/内容 | 当前来源 | 建议 |
|------|------------|----------|------|
| 路由 | `slug` | 前端用 `model_code` 转成（如 `ly-8801`） | 可选：后端加 `slug` 更灵活 |
| 图集 | `image` / `images[]` | CMS 多图 | 保持 |
| 角标 | `is_new`, `featured`(is_hot) | CMS | 保持 |
| Hero | `name`, `model_code`, `safety_standard`, `industries` | CMS | 保持 |
| 描述 | `description` | CMS 长文本 | 保持 |
| 规格-材质 | `materials.upper/toe_cap/midsole/outsole/lining` | CMS 组件 | 保持 |
| 规格-安全等级 | `safety_standard` | CMS 枚举 | 保持 |
| 规格-重量 | — | 前端写死默认文案 | **新增** `weight`（可选） |
| 规格-测试标准 | — | 前端写死默认文案 | **新增** `test_standard`（可选） |
| 规格-颜色 | — | 前端写死默认文案 | **新增** `colors`（可选） |
| 规格-尺码 | — | 前端写死 `36–48 (EU)` | **新增** `sizes`（可选） |
| 卖点标签 | `features` | CMS 多对多 Feature | 保持 |
| 行业 | `industries` | CMS 多对多 Industry | 保持 |
| 款式 | `style` | CMS 枚举 | 保持 |
| 附加认证 | `additional_certs` | CMS 多对多 Certification | 保持 |
| 业务 | `moq`, `price_range` | CMS | 保持 |

结论：**绝大部分已有；只需在现有 Product 上增加 4 个可选字段**（weight、test_standard、colors、sizes），并可选增加 `slug`，即可与详情页完全对齐，无需重做新表。

---

## 二、方案对比

### 方案 A：在现有 Product 上增量优化（推荐）

- **做法**：不新建表，只在当前 `api::product.product` 和组件 `product.material-spec` 上增字段。
- **优点**：改动小、不涉及迁移、i18n/relation 不变、前端只改 `transformProduct` 和详情页读取。
- **新增字段**（全部可选，便于旧数据兼容）：

| 字段 | 类型 | 说明 | i18n |
|------|------|------|------|
| `slug` | string | 详情页 URL 路径，如 `ly-8801`；不填则前端继续用 model_code 生成 | 否 |
| `weight` | string | 如 "As per standard" / "约 0.8 kg/只" | 是 |
| `test_standard` | string | 如 "EN ISO 20345" | 是 |
| `colors` | string | 如 "Black, Navy, Brown" | 是 |
| `sizes` | string | 如 "36–48 (EU)" | 是 |

- **material-spec**：已包含 upper / toe_cap / midsole / outsole / lining，无需改。

### 方案 B：重新做一张「产品详情」表

- **做法**：新建 collection 如 `product-detail`，与现有 `product` 一对一或通过 relation 关联；详情页专用字段放在新表。
- **优点**：产品「列表/卡片」与「详情页」模型分离，结构更清晰。
- **缺点**：需要维护两套内容、关联与迁移逻辑，对当前需求来说复杂度偏高。
- **建议**：仅当未来有「一个产品多套详情」或「详情与列表字段差异很大」时再考虑。

---

## 三、推荐实施步骤（方案 A）

### 1. 后端 Strapi：Product 增加字段

在 `safetyshoe-backend/src/api/product/content-types/product/schema.json` 的 `attributes` 中增加：

```json
"slug": {
  "type": "string",
  "unique": true,
  "pluginOptions": { "i18n": { "localized": false } }
},
"weight": {
  "type": "string",
  "pluginOptions": { "i18n": { "localized": true } }
},
"test_standard": {
  "type": "string",
  "pluginOptions": { "i18n": { "localized": true } }
},
"colors": {
  "type": "string",
  "pluginOptions": { "i18n": { "localized": true } }
},
"sizes": {
  "type": "string",
  "pluginOptions": { "i18n": { "localized": true } }
}
```

- `slug`：建议唯一，用于详情页路由；若暂不启用可先不加，前端继续用 model_code 生成。
- 其余 4 个：可选，用于详情页规格区；空则前端继续用现有默认文案。

### 2. 前端：类型与拉取

- 在 `safetyshoe-frontend/src/types/index.ts` 的 `Product` 中增加可选：  
  `slug?`, `weight?`, `test_standard?`, `colors?`, `sizes?`  
  （若保留「slug 由 model_code 生成」则 `slug` 可仍仅在前端计算。）
- 在 `safetyshoe-frontend/src/lib/strapi.ts` 的 `StrapiProductAttributes` / `StrapiProduct` 和 `transformProduct` 中映射上述字段到前端 `Product`。

### 3. 前端：详情页规格区

- 在 `ProductDetailClient` 的规格表（03）中：
  - **重量**：优先 `product.weight`，空则 `tDetail('specs.weightDefault')`。
  - **测试标准**：优先 `product.test_standard`，空则 `tDetail('specs.testDefault')`。
  - **颜色**：优先 `product.colors`，空则 `tDetail('specs.colorsDefault')`。
  - **尺码**：优先 `product.sizes`，空则写死 `36–48 (EU)` 或再做 i18n。
- 若启用后端 `slug`：详情页路由仍为 `/products/[slug]`，`page.tsx` 中由 `slug` 查产品时优先用 API 返回的 `product.slug` 匹配，无则回退到 `model_code` 转 slug。

### 4. 后台与数据

- 在 Strapi Admin 中为 Product 配置好新字段的展示（表单、列表）。
- 已有产品：新字段可为空，前端行为与现在一致；逐步在后台补全 weight/test_standard/colors/sizes 即可。

## 四、字段与详情区对应关系小结

| 详情区 | 后端字段（方案 A） | 说明 |
|--------|--------------------|------|
| 路由 | `slug`（可选） | 不填则继续用 model_code 生成 |
| Hero / 列表 | name, model_code, safety_standard, industries, style, is_hot, is_new | 已有 |
| 描述 | description | 已有 |
| 规格-材质/等级 | materials.*, safety_standard | 已有 |
| 规格-重量/标准/颜色/尺码 | weight, test_standard, colors, sizes | **新增**，可选 |
| 卖点标签 | features (relation) | 已有 |
| 附加认证 | additional_certs (relation) | 已有 |
| 业务 | moq, price_range | 已有 |
| 图集 | images (media) | 已有 |

按方案 A 做增量优化后，**产品详情页所需内容均可由 Strapi 产品表 + 现有 relation/组件 提供**，无需新表；若后续有更强「详情与列表分离」需求，再考虑方案 B 或单独详情表。
